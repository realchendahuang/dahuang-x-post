# Distribution Loop

X 文案不是发完就结束。

一条好帖要有发布后动作：

- 评论区承接链接、证据、补充。
- 回复早期读者，把评论变成内容分支。
- 爆帖拆新角度，一条内容发不止一次。

## 1. 主帖和首评分工

主帖负责：

- 抓人。
- 给判断。
- 给一个具体证据。
- 让人想回复或点开。

首评负责：

- 链接。
- GitHub 仓库。
- Demo。
- 长文。
- 截图补充。
- 代码片段。
- 更长解释。

例：

```md
## 主帖
我现在发开源项目，很少第一句贴仓库。

先讲它把什么问题变简单。
再把仓库放首评。

人先被问题抓住，才会关心链接。

## 首评
仓库在这里：
{url}

我也放了一个 40 秒 Demo，能看到改前/改后的差别。
```

## 2. 外链策略

不要让外链成为主帖主角。

更稳的写法：

1. 主帖讲核心判断/结果。
2. 用图片、截图、Thread、短视频等原生内容先承载价值。
3. 链接放首评或最后补充。
4. 如果必须主帖带链接，第一行仍然先写判断，不要第一眼就是 URL。

判断主帖该不该带链接，问一句：

`这条去掉链接后，还值不值得看？`

不值得，先改主帖，别靠链接救场。

配图/截图帖同理：第一行仍写判断，不写"看图"；图里别堆字，X 上图字多了没人读；文案点出图里最关键的一个变化（改前/改后、一个数字），代替复述全图。

## 3. 评论区承接模板

### 补链接

```md
补链接：
{url}

建议先看 {具体部分}，这部分最能说明问题。
```

### 补案例

```md
补一个具体例子：

改前用户要 {步骤数} 步。
改后只剩 {步骤数} 步。
```

### 补截图

```md
截图里看得最明显：

不是界面变漂亮了。
是用户不用再猜下一步点哪里。
```

### 补延伸观点

```md
补一句：

这类内容别急着写教程。
先把"为什么值得关心"讲清楚。
```

## 4. 回复策略

回复不是寒暄。

高质量回复至少做一件事：

- 补一个事实。
- 给一个例子。
- 承认一个边界。
- 把对方问题变成新帖角度。
- 顺手把自己相关内容挂回去，但不要硬广。

弱：

`感谢支持！`

强：

`对，这个边界很重要。链接放首评不是永远正确，如果账号是 Premium 且目标就是点击，主帖带链接也可以测。`

## 5. 爆帖二次利用

如果一条跑起来，立刻做 5 件事：

1. 在原帖评论区补链接/截图/新证据。
2. 回复高质量评论，把它们变成内容分支。
3. 拆一个争议点写新帖。
4. 拆一个方法步骤写 Thread。
5. 把新帖贴回原帖评论区。

例：

主帖爆点：

`项目更新不要写成公告。`

可拆新角度：

- `发开源项目时，第一句别写什么。`
- `一个 GitHub 仓库如何拆 5 条 X 内容。`
- `为什么链接应该后置。`
- `项目截图怎么写说明文字。`
- `收到用户反馈后怎么二次发帖。`

原则只有一句：

不要复读原帖。

每次复用都要换读者、场景、动作、证据。

## 6. 看数据决定下一轮

复盘时，别只看点赞：

- Views 高、回复低：hook 能抓人，正文不够可讨论。
- 回复高、主页点击低：有话题，但个人品牌线索弱。
- 收藏高：有工具性，适合拆 Thread 或长文。
- 链接点击高、传播低：资源有用，但主帖表达弱。
- Quote 多：观点有冲突，补边界避免误读。

发布时间、链接位置、媒体形式，都要结合账号数据测，不要照搬别人的"最佳时间"。
