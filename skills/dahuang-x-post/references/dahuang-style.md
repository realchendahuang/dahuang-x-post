# Dahuang Style Rules

这份规则负责中文 X 语感。

外部资料给结构，大黄规则给味道。

## 一句话核心

像一个真的做过事的人，在信息流里直接复盘。

不要像公众号。

不要像课堂。

不要像 AI 在总结方法论。

## 1. 观点前置

第一句话先亮判断。

不要先铺背景。

不要先说：

- `我最近有个感受`
- `最近很多朋友问我`
- `在当下这个时代`
- `今天想聊聊`
- `先说结论`

直接说：

- `中文 X 最浪费的一类内容，是把项目更新写成公告。`
- `我现在越来越确定，评论区不是补充，是第二条内容。`
- `发开源项目，第一句贴仓库通常是下策。`

## 2. 一行一句

一行只承载一个信息：

- 判断。
- 理由。
- 例子。
- 转折。
- 动作。

如果一行里有两个意思，拆。

如果一个段落超过 3 行，拆。

如果拆完仍然读起来像文章，重排，不要继续润色。

## 3. 短句高密度

短，不等于空。

每 2-3 行至少要有一个具体东西：

- 数字。
- 项目名。
- 时间。
- 用户原话。
- 截图说明。
- 操作步骤。
- 对比。
- 失败教训。

弱：

`这件事很重要，值得每个创作者重视。`

强：

`我只换了第一行，评论从"支持"变成了具体提问。`

## 4. 去爹味

不要教育别人。

不要写：

- `你必须`
- `你应该`
- `大家一定要`
- `只有...才能`

改成复盘：

弱：

`你必须持续输出，才能获得更多曝光。`

强：

`我后来发现，断更最伤的不是曝光，是别人忘了你在做什么。`

## 5. 去车轱辘话

同一个意思出现两次，删一次。

解释比判断还长，砍解释。

一句话没有新信息，删。

判断站不住，不要靠排比撑气势。

## 6. 禁用壳

默认删：

- `不是...而是...`
- `不只是...更是...`
- `本质上`
- `值得注意的是`
- `随着...不断发展`
- `在当下这个时代`
- `让我们一起`
- `未来可期`
- `综上所述`
- `你怎么看？`
- `欢迎在评论区留言`
- `省流版`
- `一句话总结`

替换方法：

- `不是 A 而是 B` -> 直接说 B，A 如果有价值另起一行。
- `值得注意的是 X` -> 直接说 X。
- `你怎么看` -> 问一个具体问题。

## 7. 防 AI 腔

X 帖子最常踩的 AI 腔不在词汇，在**组织方式**。单个口癖不判 AI 味——集群才定罪。一条帖里叠两三个，就是 AI 在表演写作。

### 句式壳（最要命）

X 帖子最常踩的三种 AI 句式：

- **二分对照壳**：`不是 A，而是 B` / `不在于 A，而在于 B` / `与其说 A，不如说 B`。改法：把 B 直接说清楚，A 有信息就换普通句式另起一行，多余就删。
- **本质拔高壳**：`真正重要的是` / `真正决定 X 的是` / `核心在于` / `底层逻辑`。改法：删掉"真正""本质""核心"，直接说结论。
- **否定式排比**：`这不仅仅是 X，更是 Y` / `不只是 X，更是 Y`。改法：删掉否定式，直接说它是什么。

> `不是 A 而是 B` 是跨风格 AI 通用句式，所有模型都用。建议零容忍——除非刻意用且有理由，否则一律改。

### 词汇壳

除了第 6 节的禁用壳，这些也是 X 帖子高频 AI 词汇：

- **路标词**：`接下来我们` / `下面我们` / `由此可见` / `换句话说` / `简而言之` / `归根结底` / `不难发现` / `诚然` —— 删掉，直接说那件事。
- **伪口语动作词**：`拆解` / `梳理` / `剖析` / `盘点` / `聚焦` / `洞察` —— 删掉动作词，直接讲内容。"我们来拆解一下这个问题" → 直接分析问题。
- **互联网黑话**：`赋能` / `助力` / `打造` / `抓手` / `闭环` / `沉淀` / `痛点` / `降本增效` / `链路` / `触达` —— 换成具体做了什么。"赋能开发者" → "让开发者能……"。技术语境里如果是真行话可留。
- **渲染性强调**：`深刻的` / `深远的` / `至关重要` / `举足轻重` / `令人瞩目` / `意义非凡` / `前所未有` / `颠覆性` —— 删掉渲染词，留事实。
- **无源引用**：`研究表明` / `数据显示` / `有专家指出` / `业内普遍认为` —— 删掉模糊归因，直接给结论；有具体来源就换成具体引用。
- **思维方法论腔**：`底层逻辑` / `第一性原理` / `认知升级` / `心智模型` / `长期主义` / `复利思维` —— 换成具体在说什么，不要用一个框架词打包。

### 交流壳

- **假互动结尾**：`你觉得呢？` / `是不是很有启发？` / `你有没有类似经历？` —— 删掉。X 帖子要互动就问具体问题（见第 6 节），不要泛泛求互动。
- **谄媚腔**：`好问题！` / `这个思路简直绝了` / `你的观察力太敏锐了` —— 全部删掉，直接回应内容。
- **协作痕迹**：`希望这能帮到你` / `当然！` / `请告诉我` —— 正文不是聊天框，删掉。

### 删壳不删判断

删的是壳，不是判断。最小改写：

- 能局部改就不整段重写。删掉模板句式，把信息塞回叙述流。
- 不做机械同义词替换。把"深入探讨"换成"讨论"不够——要把整句结构改掉，让"深入探讨"这个位置不再需要存在。
- 删完读一遍，如果信息密度下降、作者锋利被抹平了，回滚——宁可保留一点锋利，也不要改成通用的"干净中性句"。

### 怎么扫

交付前自问一句：**这条帖子为什么一看就是 AI 写的？** 列出来还剩什么壳，再改一遍。

集群判断：一条帖里同一个层的壳出现 ≥2 个，就是 AI 在表演；偶尔一个口癖不判罪，叠一起才是。

## 8. 不滥用排比

三连排比容易变 AI 味。

如果连续三行同构，打断其中一行。

弱：

```md
要持续输出。
要持续互动。
要持续复盘。
```

强：

```md
持续输出只是第一层。

更关键的是：
别人能不能反复看到你在同一个方向上出现。
```

## 9. 中英混排

中文 X 创作者几乎必混英文：产品名、术语、命令、项目名。

原则：

- 产品名、命令、项目名保留原文，不硬翻译。
- 不为了显得专业堆英文术语。
- 一句话里中英混到读不顺时，拆。

弱：

`我用 Claude Code 这个 AI coding 的 agent 做了一次 codebase 的 refactor。`

强：

`我用 Claude Code 重构了一个模块。

省了大概两小时。`

中英文之间的空格不强求——X 上随意——但同句混排别到读不顺。

## 10. 中文 X 语感

可以直接。

可以有点怪。

可以锋利。

可以说人话。

但不要为了显得会写，牺牲信息密度。

最终读起来要像：

`这个人真的做过，真的踩过坑，现在把最有用的一句拎出来了。`

## 边界

这份只管语感（味道、句式、用词）。

不管：

- 正文结构和 Hook 矩阵 → `references/viral-post-playbook.md`
- Thread 形态 → `references/thread-playbook.md`
- 平台字符/媒体限制 → `references/platform-rules.md`
